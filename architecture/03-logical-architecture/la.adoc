== Contexte et objectif

L'Architecture Logique (LA) définit la *décomposition interne du système* en composants logiques cohérents, leurs responsabilités, leurs interfaces et leurs interactions. Elle répond à la question : *comment le système est-il structuré pour remplir les fonctions identifiées en SA ?*

C'est à ce niveau que l'on travaille la *logique interne du système* : enchaînements de traitements, règles métier, gestion des états et coordination entre composants — indépendamment de toute technologie d'implémentation.

La LA ne choisit pas encore de technologies d'implémentation (langages, frameworks, bases de données) — c'est le rôle de la PA (Physical Architecture). Elle pose les *frontières architecturales stables* sur lesquelles la PA s'appuiera.

NOTE: Chaque composant logique de la LA est tracé vers une ou plusieurs fonctions système (FS) de la SA. Les interfaces logiques (IFL) précisent les contrats d'échange entre composants, indépendamment du protocole de transport retenu en PA.

=== Principes architecturaux retenus

* *Séparation des responsabilités* — chaque composant a une responsabilité unique et cohérente.
* *Couplage faible* — les composants communiquent via des interfaces contractuelles explicites (IFL).
* *Traçabilité SA → LA* — toute fonction système (FS) est allouée à exactement un composant logique.
* *Isolation des systèmes externes* — les intégrations tierces (paiement, transporteur, messagerie) sont encapsulées dans des composants dédiés de type *adaptateur*.

== Allocation des fonctions système aux composants logiques

Le système est structuré en cinq couches logiques : *Présentation*, *Application*, *Domaine*, *Infrastructure* et *Intégrations externes*.

== Activation ==
Buyer -> UI : GET /activate?token={uuid}
UI -> SvcAccount : IFL-01 — activerCompte(token)

  SvcAccount --> UI : 200 OK
  UI --> Buyer : Compte activé — redirection connexion
else Token expiré
  SvcAccount --> UI : 410 Gone — token expiré
  UI --> Buyer : Proposer le renvoi du lien
  Buyer -> UI : Demander le renvoi du lien
  UI -> SvcAccount : IFL-01 — renvoyerLienActivation(email)

  SvcNotif -> AdpMail : IFL-12 — envoyerEmail(destinataire, template:activation, {lienToken})
  AdpMail -> ExtMail : IFS-06 — SMTP / API POST
  note right : Reprend le flux d'acheminement\ndu lien d'activation
else Token déjà utilisé
  SvcAccount --> UI : 409 Conflict — compte déjà activé
  UI --> Buyer : "Votre compte est déjà actif"
end

== Modèle de données logique — Compléments LA

La SA a défini les entités métier. La LA précise les entités techniques nécessaires au fonctionnement des composants logiques.

package "Entités métier (SA)" #F0F8FF {

  class CompteUtilisateur {
    + id : UUID
    + email : String
    + motDePasseHaché : String
    + rôle : Enum {admin, vendeur, acheteur}
    + statut : Enum {en_attente, actif, suspendu, supprimé}
    + langue : Enum {fr, en}
    + dateCreation : DateTime
  }

  class AdresseLivraison {
    + id : UUID
    + compte : CompteUtilisateur
    + libellé : String
    + rue : String
    + ville : String
    + codePostal : String
    + paysZoneEuro : CodeISO
    + estParDefaut : Boolean
    + supprimé : Boolean
  }
  note right of AdresseLivraison
    FS-B11 / US-PRF-03
    `compte` est une référence directe à CompteUtilisateur
    (clé étrangère account_id en base).
    Suppression logique (supprimé = true) préservant
    l'intégrité référentielle des commandes passées.
    Une seule adresse par compte peut avoir estParDefaut = true.
    Minimum 1 adresse active par compte (contrainte service).
  end note

  class Produit {
    + id : UUID
    + nom : String
    + description : Text
    + prix : Decimal
    + photos : List<URL>
    + catégorie : String
    + statut : Enum {publié, archivé}
    + quantitéDisponible : Integer
    + seuilAlerte : Integer
  }

  class Panier {
    + id : UUID
    + dateMiseAJour : DateTime
  }

  class LignePanier {
    + quantité : Integer
    + prixUnitaireCapturé : Decimal
  }

  class Commande {
    + id : UUID
    + adresseLivraisonId : UUID
    + modeLivraison : String
    + montantTotal : Decimal
    + statut : Enum {paiement_en_attente_virement,\nen_attente, en_préparation,\nexpédiée, demande_annulation,\nen_attente_retour,\nremboursement_virement_en_cours,\nlivrée, annulée}
    + dateCommande : DateTime
  }

  class LigneCommande {
    + quantité : Integer
    + prixUnitaireCapturé : Decimal
  }

  class Paiement {
    + id : UUID
    + montant : Decimal
    + mode : Enum {carte, virement}
    + statut : Enum {en_attente_virement,\nautorisé, refusé, remboursé,\nvirement_inverse_en_cours}
    + référencePrestataire : String
    + ibanAcheteur : String
    + date : DateTime
  }

  class Expédition {
    + id : UUID
    + numéroSuivi : String
    + statutLivraison : String
    + dateExpédition : DateTime
    + dateLivraisonEstimée : Date
  }

  class Transporteur {
    + id : UUID
    + nom : String
    + paysSupportés : List<CodeISO>
    + urlSuivi : URL
    + actif : Boolean
  }

}
