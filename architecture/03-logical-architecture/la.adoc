== Contexte et objectif

L'Architecture Logique (LA) définit la *décomposition interne du système* en composants logiques cohérents, leurs responsabilités, leurs interfaces et leurs interactions. Elle répond à la question : *comment le système est-il structuré pour remplir les fonctions identifiées en SA ?*

La LA ne choisit pas encore de technologies d'implémentation (langages, frameworks, bases de données) — c'est le rôle de la PA (Physical Architecture). Elle pose les *frontières architecturales stables* sur lesquelles la PA s'appuiera.

NOTE: Chaque composant logique de la LA est tracé vers une ou plusieurs fonctions système (FS) de la SA. Les interfaces logiques (IFL) précisent les contrats d'échange entre composants, indépendamment du protocole de transport retenu en PA.

=== Principes architecturaux retenus

* *Séparation des responsabilités* — chaque composant a une responsabilité unique et cohérente.
* *Couplage faible* — les composants communiquent via des interfaces contractuelles explicites (IFL).
* *Traçabilité SA → LA* — toute fonction système (FS) est allouée à exactement un composant logique.
* *Isolation des systèmes externes* — les intégrations tierces (paiement, transporteur, messagerie) sont encapsulées dans des composants dédiés de type *adaptateur*.

== Décomposition en composants logiques

Le système est structuré en cinq couches logiques : *Présentation*, *Application*, *Domaine*, *Infrastructure* et *Intégrations externes*.

== Interfaces logiques (IFL)

Les interfaces logiques définissent les contrats d'échange entre composants, indépendamment du protocole technique retenu en PA.

[cols="1,1,1,2,1", options="header"]
|===
| Identifiant | Émetteur | Récepteur | Données / opérations | FS SA couvertes

| *IFL-01*
| Portails (UI)
| Service Compte
| Inscription, activation token, authentification, mise à jour profil, gestion rôles, gestion transporteurs
| FS-B01, FS-B02, FS-B10, FS-A01..A03

| *IFL-02*
| Portails (UI)
| Service Catalogue
| Lister produits, rechercher, filtrer, CRUD produits et catégories
| FS-B03, FS-V02

| *IFL-03*
| Portail Acheteur
| Service Panier
| Ajouter / modifier / supprimer une ligne, obtenir le panier courant, restaurer le panier sauvegardé
| FS-B04, FS-B05

| *IFL-04*
| Portails (UI)
| Service Commande
| Créer une commande, lister les commandes, mettre à jour le statut, demander une annulation
| FS-B06, FS-B09, FS-V04, FS-V08

| *IFL-05*
| Portails (UI) / Service Commande
| Service Expédition
| Déclarer une expédition, obtenir le suivi, recevoir les mises à jour de statut
| FS-V05, FS-C01, FS-C02, FS-B08

| *IFL-06*
| Service Commande / Portail Vendeur
| Service Stock
| Réserver du stock, libérer du stock, interroger les niveaux, déclencher une alerte rupture
| FS-V03

| *IFL-07*
| Back-office Vendeur
| Service Rapport
| Obtenir les agrégats de ventes, l'historique, les statistiques produits
| FS-V07

| *IFL-08*
| Service Commande
| Service Paiement
| Demander une autorisation, demander un remboursement, interroger le statut d'un paiement
| FS-P01, FS-P02, FS-P03

| *IFL-09*
| Services (Commande, Compte, Paiement, Expédition)
| Service Notification
| Émettre une notification (type, destinataire, données variables)
| Toutes notifications

| *IFL-10*
| Service Paiement
| Adaptateur Paiement
| Autoriser transaction, rembourser, interroger statut — traduit vers IFS-04
| FS-P01, FS-P02, FS-P03

| *IFL-11*
| Service Expédition
| Adaptateur Transporteur
| Créer un envoi, recevoir un webhook de statut — traduit vers IFS-05
| FS-C01, FS-C02

| *IFL-12*
| Service Notification
| Adaptateur Messagerie
| Envoyer un email (destinataire, template, variables) — traduit vers IFS-06
| FS-B01, toutes notifications email

| *IFL-13*
| Services applicatifs
| Référentiel de données
| CRUD sur toutes les entités du modèle de données logique (SA §5)
| Support de toutes les FS
|===

Ce diagramme illustre les interactions entre composants logiques lors du flux d'inscription et d'activation de compte acheteur (UCSA-01).

----
skinparam sequenceArrowColor #2E75B6

Buyer -> UI : Soumettre le formulaire d'inscription
UI -> SvcAccount : IFL-01 — creerCompte(email, mdpHashé, profil)
SvcAccount -> DB : IFL-13 — INSERT compte {statut: en_attente}
SvcAccount -> SvcAccount : Générer token UUID + expiration 24h (CS-07)
SvcAccount -> DB : IFL-13 — INSERT token_activation {uuid, compte_id, expireAt}
SvcAccount -> SvcNotif : IFL-09 — notifier(ACTIVATION_EMAIL, email, {token})
SvcNotif -> AdpMail : IFL-12 — envoyerEmail(destinataire, template:activation, {lienToken})
AdpMail -> ExtMail : IFS-06 — SMTP / API POST
ExtMail --> Buyer : Email — lien d'activation

SvcAccount --> UI : 201 Created — compte en attente d'activation
UI --> Buyer : "Vérifiez votre boîte mail"

== Activation ==
Buyer -> UI : GET /activate?token={uuid}
UI -> SvcAccount : IFL-01 — activerCompte(token)
SvcAccount -> DB : IFL-13 — SELECT token WHERE uuid = {uuid}

alt Token valide et non expiré
  SvcAccount -> DB : IFL-13 — UPDATE compte {statut: actif}
  SvcAccount -> DB : IFL-13 — DELETE token_activation
  SvcAccount --> UI : 200 OK
  UI --> Buyer : Compte activé — redirection connexion
else Token expiré
  SvcAccount --> UI : 410 Gone — token expiré
  UI --> Buyer : Proposer le renvoi du lien
else Token déjà utilisé
  SvcAccount --> UI : 409 Conflict — compte déjà activé
  UI --> Buyer : "Votre compte est déjà actif"
end
----
