== Contexte et objectif

L'Architecture Physique (PA) répond à la question : *quelles technologies et quelle infrastructure concrète implémentent l'architecture logique ?* Elle mappe chaque composant logique de la LA sur un composant physique réel — module de code, conteneur, service cloud — et définit les contraintes de déploiement, de sécurité et de scalabilité.

La PA s'appuie directement sur la LA : les composants logiques identifiés sont affectés à des nœuds de déploiement et des technologies choisies. Les interfaces logiques (IFL) se traduisent ici en protocoles et APIs concrètes.

NOTE: Les choix technologiques retenus ci-dessous sont des recommandations architecturales. Ils peuvent être substitués par des équivalents fonctionnels sans remettre en cause les frontières logiques définies en LA.

== Contraintes physiques (CPA)

Les contraintes physiques complètent les contraintes système (CS) en ajoutant les exigences d'infrastructure et de déploiement.

[cols="1,1,3", options="header"]
|===
| Identifiant | Contrainte physique | Description

| CPA-01
| Déploiement conteneurisé
| Chaque composant déployable est encapsulé dans un conteneur Docker. L'orchestration locale utilise Docker Compose (développement) et Kubernetes (production).

| CPA-02
| HTTPS obligatoire
| Toutes les communications réseau utilisent TLS 1.2 minimum. La terminaison TLS est assurée par le reverse proxy (Nginx).

| CPA-03
| Transactions ACID
| La base de données relationnelle (PostgreSQL) garantit l'intégrité des opérations critiques — commandes, paiements, réservations de stock.

| CPA-04
| Cache distribué pour les sessions
| La persistance du panier (CS-03) et la gestion des sessions utilisateur sont déportées dans Redis, indépendamment du serveur d'application (scalabilité horizontale possible).

| CPA-05
| Serveur d'application stateless
| Le serveur d'application ne stocke aucun état en mémoire locale. Tout l'état métier est externalisé dans PostgreSQL ou Redis.

| CPA-06
| Environnements séparés
| Trois environnements distincts : développement (local Docker Compose), staging (pré-production), production.

| CPA-07
| Journalisation des transactions financières
| Toutes les transactions (paiement, remboursement) sont journalisées en base avec horodatage, montant, statut et référence prestataire — issu de CS-06.

| CPA-08
| Implémentation de la TokenAuthentification (CS-08)
| Les jetons sont signés avec HMAC-SHA-256 (JJWT), ont une validité de 24 h et sont transmis dans l'en-tête `Authorization: Bearer`. Les mots de passe sont hachés avec BCrypt (coût 10). La clé de signature est injectée par variable d'environnement (`JWT_SECRET`). Les jetons révoqués sont inscrits dans Redis (CPA-04).
|===

participant "SendGrid\n(IFS-06)" as Mail

API -> Redis : GET cart:{customer_id}

alt Card payment
  API -> CatalogMod : reserveStock(lines[])
  CatalogMod -> PG : BEGIN ; UPDATE stock SET qty = qty - n WHERE id = ?
  PG --> CatalogMod : OK
  CatalogMod --> API : stock reserved

  API -> PG : INSERT INTO orders {status: pending, ...}
  PG --> API : order_id

  API -> PayMod : submitPayment(amount, mode:card, order_id)

  note right : Stripe Java SDK\nIFS-04

  alt Payment authorized
    Stripe --> PayMod : 200 OK {payment_intent_id, status: "succeeded"}
    PayMod -> PG : INSERT INTO payments {status: authorized, stripe_ref, ...}\n(CPA-07 — CS-06)
    PG --> PayMod : OK
    PayMod --> API : payment confirmed

    API -> PG : UPDATE orders SET status = pending WHERE id = order_id
    PG --> API : OK

    API -> NotifMod : notify(customer_id, "order_confirmed", order_id)
    NotifMod -> Mail : POST /v3/mail/send\n{to, template_id: "order_confirmation", ...}
    note right : SendGrid Java SDK\nIFS-06
    Mail --> NotifMod : 202 Accepted

    API -> NotifMod : notify(vendor_id, "new_order", order_id)
    NotifMod -> Mail : POST /v3/mail/send {to: vendor, ...}
    Mail --> NotifMod : 202 Accepted

    API --> SPA : 201 Created {order_id, status: "pending"}

  else Payment declined
    Stripe --> PayMod : 402 {error.code, error.message}
    PayMod -> PG : INSERT INTO payments {status: declined, ...}
    PayMod -> CatalogMod : releaseStock(lines[])
    CatalogMod -> PG : ROLLBACK stock reservation
    PayMod --> API : payment declined
    API -> PG : DELETE FROM orders WHERE id = order_id
    API --> SPA : 402 {reason: error.message}
  end

else Wire transfer payment (CS-02 / FS-V09)
  API -> CatalogMod : preReserveStock(lines[])
  CatalogMod -> PG : BEGIN ; LOCK stock ; SET qty_reserved = qty_reserved + n
  PG --> CatalogMod : OK
  CatalogMod --> API : stock pre-reserved

  API -> PG : INSERT INTO orders {status: pending_wire_transfer, ...}
  PG --> API : order_id

  API -> PayMod : registerPendingTransfer(amount, order_id)
  PayMod -> PG : INSERT INTO payments {status: pending_wire_transfer, mode: wire, ...}\n(CPA-07 — CS-06)
  PG --> PayMod : OK
  PayMod --> API : transfer registered pending

  API -> NotifMod : notify(customer_id, "wire_transfer_pending", {bank_details})
  NotifMod -> Mail : POST /v3/mail/send {to, template_id: "wire_transfer_pending", ...}
  Mail --> NotifMod : 202 Accepted

  API -> NotifMod : notify(vendor_id, "wire_transfer_to_confirm", order_id)
  NotifMod -> Mail : POST /v3/mail/send {to: vendor, template_id: "wire_transfer_to_confirm", ...}
  Mail --> NotifMod : 202 Accepted

  API --> SPA : 201 Created {order_id, status: "pending_wire_transfer"}

  == Wire transfer confirmation by vendor ==
  participant "SPA React\n(vendor back-office)" as SPAVendor
  SPAVendor -> API : POST /orders/{id}/confirm-transfer\nAuthorization: Bearer {JWT vendor}

  API -> PayMod : confirmTransfer(order_id)
  PayMod -> PG : UPDATE payments SET status = authorized WHERE order_id = ?
  PG --> PayMod : OK
  PayMod --> API : transfer confirmed

  API -> PG : UPDATE orders SET status = pending WHERE id = order_id
  PG --> API : OK

  API -> NotifMod : notify(customer_id, "order_confirmed", order_id)
  NotifMod -> Mail : POST /v3/mail/send {to, template_id: "order_confirmation", ...}
  Mail --> NotifMod : 202 Accepted

  API -> NotifMod : notify(vendor_id, "new_order", order_id)
  NotifMod -> Mail : POST /v3/mail/send {to: vendor, ...}
  Mail --> NotifMod : 202 Accepted

  API --> SPAVendor : 200 OK {order_id, status: "pending"}
end

==== Page d'accueil

*Carrousel d'annonces* :

* Une seule annonce est visible à la fois ; le défilement est automatique (intervalle configurable) et peut être contrôlé manuellement via les flèches `<` / `>`.
* Chaque annonce peut contenir un texte, une image ou les deux, selon la configuration du vendeur.
* Si un lien est renseigné, l'ensemble du bloc annonce est cliquable et redirige l'internaute vers ce lien.
* Les indicateurs de position (● ○ ○) reflètent le nombre d'annonces et la position courante.
* Si aucune annonce n'est configurée, le carrousel est masqué et le catalogue s'affiche directement.
