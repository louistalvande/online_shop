== Contexte et objectif

L'Architecture Physique (PA) répond à la question : *quelles technologies et quelle infrastructure concrète implémentent l'architecture logique ?* Elle mappe chaque composant logique de la LA sur un composant physique réel — module de code, conteneur, service cloud — et définit les contraintes de déploiement, de sécurité et de scalabilité.

La PA s'appuie directement sur la LA : les seize composants logiques identifiés sont affectés à des nœuds de déploiement et des technologies choisies. Les interfaces logiques (IFL) se traduisent ici en protocoles et APIs concrètes.

NOTE: Les choix technologiques retenus ci-dessous sont des recommandations architecturales. Ils peuvent être substitués par des équivalents fonctionnels sans remettre en cause les frontières logiques définies en LA.

== Contraintes physiques (CPA)

Les contraintes physiques complètent les contraintes système (CS) en ajoutant les exigences d'infrastructure et de déploiement.

[cols="1,1,3", options="header"]
|===
| Identifiant | Contrainte physique | Description

| CPA-01
| Déploiement conteneurisé
| Chaque composant déployable est encapsulé dans un conteneur Docker. L'orchestration locale utilise Docker Compose (développement) et Kubernetes (production).

| CPA-02
| HTTPS obligatoire
| Toutes les communications réseau utilisent TLS 1.2 minimum. La terminaison TLS est assurée par le reverse proxy (Nginx).

| CPA-03
| Transactions ACID
| La base de données relationnelle (PostgreSQL) garantit l'intégrité des opérations critiques — commandes, paiements, réservations de stock.

| CPA-04
| Cache distribué pour les sessions
| La persistance du panier (CS-03) et la gestion des sessions utilisateur sont déportées dans Redis, indépendamment du serveur d'application (scalabilité horizontale possible).

| CPA-05
| Serveur d'application stateless
| Le serveur d'application ne stocke aucun état en mémoire locale. Tout l'état métier est externalisé dans PostgreSQL ou Redis.

| CPA-06
| Environnements séparés
| Trois environnements distincts : développement (local Docker Compose), staging (pré-production), production.

| CPA-07
| Journalisation des transactions financières
| Toutes les transactions (paiement, remboursement) sont journalisées en base avec horodatage, montant, statut et référence prestataire — issu de CS-06.
|===

== Diagramme de séquence technique — Tunnel d'achat et paiement Stripe

Ce diagramme représente les échanges techniques complets lors de la passation d'une commande, depuis la SPA React jusqu'aux systèmes externes, en faisant apparaître les protocoles, SDK et conventions API concrètes.

[plantuml, pa-sequence-commande, svg]
----
skinparam sequenceArrowColor #2E75B6
skinparam sequenceParticipantBackgroundColor #E8F4FD
skinparam sequenceParticipantBorderColor #2E75B6
skinparam sequenceLifeLineBorderColor #2E75B6
skinparam noteBorderColor #F0A500
skinparam noteBackgroundColor #FFFBE6
skinparam defaultFontSize 10

participant "SPA React\n(Nginx)" as SPA
participant "API Node.js\n(module-commande)" as API
participant "module-paiement\n(Stripe SDK)" as PayMod
participant "module-stock" as StockMod
participant "module-notification\n(SendGrid SDK)" as NotifMod
participant "PostgreSQL" as PG
participant "Redis" as Redis
participant "Stripe API\n(IFS-04)" as Stripe
participant "SendGrid\n(IFS-06)" as Mail

SPA -> API : POST /commandes\n{panier_id, adresse_UE, mode_livraison, mode_paiement}\nAuthorization: Bearer {JWT}
note right : HTTPS (CPA-02)\nCS-04 : adresse validée UE

API -> Redis : GET panier:{acheteur_id}
Redis --> API : lignes du panier

API -> StockMod : reserverStock(lignes[])
StockMod -> PG : BEGIN ; UPDATE stock SET qté = qté - n WHERE id = ?
PG --> StockMod : OK
StockMod --> API : stock réservé

API -> PG : INSERT INTO commandes {statut: en_attente, ...}
PG --> API : commande_id

API -> PayMod : soumettrePaiement(montant, mode, commande_id)
PayMod -> Stripe : POST /v1/payment_intents\n{amount, currency, payment_method, metadata}
note right : SDK Stripe Node.js\nIFS-04

alt Paiement autorisé
  Stripe --> PayMod : 200 OK {payment_intent_id, status: "succeeded"}
  PayMod -> PG : INSERT INTO paiements {statut: autorisé, ref_stripe, ...}\n(CPA-07 — journalisation CS-06)
  PG --> PayMod : OK
  PayMod --> API : paiement confirmé

  API -> PG : UPDATE commandes SET statut = en_préparation WHERE id = commande_id
  PG --> API : OK

  API -> NotifMod : notifier(acheteur_id, "commande_confirmée", commande_id)
  NotifMod -> Mail : POST /v3/mail/send\n{to, template_id: "confirmation_commande", ...}
  note right : SDK SendGrid\nIFS-06
  Mail --> NotifMod : 202 Accepted

  API -> NotifMod : notifier(vendeur_id, "nouvelle_commande", commande_id)
  NotifMod -> Mail : POST /v3/mail/send {to: vendeur, ...}
  Mail --> NotifMod : 202 Accepted

  API --> SPA : 201 Created {commande_id, statut: "en_préparation"}

else Paiement refusé
  Stripe --> PayMod : 402 {error.code, error.message}
  PayMod -> PG : INSERT INTO paiements {statut: refusé, ...}
  PayMod -> StockMod : libererStock(lignes[])
  StockMod -> PG : ROLLBACK réservation stock
  PayMod --> API : paiement refusé
  API -> PG : DELETE FROM commandes WHERE id = commande_id
  API --> SPA : 402 {raison: error.message}
end
----

API -> PG : SELECT * FROM commandes WHERE id = ? AND statut = en_préparation
PG --> API : commande

API -> CarrierAPI : POST /shipments\n{weight, dimensions, adresse_destinataire, service}
note right : REST/HTTPS (IFS-05)\nCS-04 : pays_destination dans l'UE

CarrierAPI --> API : 201 Created {tracking_number, label_url}

API -> PG : INSERT INTO expeditions {tracking_number, transporteur_id, ...}
API -> PG : UPDATE commandes SET statut = expediée WHERE id = ?
PG --> API : OK

API -> NotifMod : notifier(acheteur_id, "commande_expediee", tracking_number)
NotifMod -> Mail : POST /v3/mail/send\n{to, template_id: "expedition", tracking_number}
Mail --> NotifMod : 202 Accepted

API --> SPA : 200 OK {tracking_number}

== Webhooks de statut (asynchrone) ==

CarrierAPI -> API : POST /webhooks/tracking\n{tracking_number, statut, timestamp, localisation}
note right : Appel entrant — IFS-05\nNginx forward vers API

API -> PG : INSERT INTO statuts_acheminement\n{tracking_number, statut, timestamp}
PG --> API : OK

API -> NotifMod : notifier(acheteur_id, "statut_livraison", statut)
NotifMod -> Mail : POST /v3/mail/send\n{to, template_id: "statut_livraison", statut}
Mail --> NotifMod : 202 Accepted

API --> CarrierAPI : 200 OK
